# Coding Conventions

**Analysis Date:** 2026-03-03

## Naming Patterns

**Files:**
- Use `snake_case.rs` for all Rust source files: `app_tests.rs`, `shell_popup.rs`, `worktree.rs`
- Module entry points are `mod.rs` within their directory: `src/db/mod.rs`, `src/git/mod.rs`
- Test files in `tests/` are named `{module}_tests.rs`: `db_tests.rs`, `git_tests.rs`, `board_tests.rs`
- Plugin config files are `plugin.toml` inside a named directory: `plugins/gsd/plugin.toml`
- Skill files are plain markdown with YAML frontmatter: `plugins/agtx/skills/plan.md`

**Functions:**
- Use `snake_case` for all functions: `create_worktree()`, `build_interactive_command()`, `handle_normal_key()`
- Prefix getters with `get_`: `get_task()`, `get_all_tasks()`, `get_all_projects()`
- Prefix checkers with `is_`/`has_`: `is_available()`, `is_at_bottom()`, `has_changes()`, `has_session()`
- Boolean-returning functions use predicate names: `worktree_exists()`, `session_exists()`, `supports_agent()`
- Drawing functions use `draw_` prefix and are static methods: `draw_board()`, `draw_shell_popup()`, `draw_task_card()`
- Key handlers use `handle_` prefix: `handle_key()`, `handle_normal_key()`, `handle_shell_popup_key()`
- Constructors use `new()` or `open_*()`: `Task::new()`, `Database::open_project()`, `Database::open_global()`

**Variables:**
- Use `snake_case` for all variables: `worktree_path`, `task_slug`, `branch_name`
- Use `_` prefix for intentionally unused variables: `let _ = result;`

**Types:**
- Use `PascalCase` for all types: `TaskStatus`, `BoardState`, `ShellPopup`, `MergedConfig`
- Enum variants are `PascalCase`: `TaskStatus::Backlog`, `InputMode::Normal`, `PhaseStatus::Working`
- Trait names describe capabilities: `GitOperations`, `TmuxOperations`, `AgentOperations`, `AgentRegistry`
- Real implementations prefixed with `Real`: `RealGitOps`, `RealTmuxOps`, `RealGitHubOps`, `RealAgentRegistry`
- Mock implementations prefixed with `Mock`: `MockGitOperations`, `MockTmuxOperations` (auto-generated by `mockall`)

**Constants:**
- Use `SCREAMING_SNAKE_CASE`: `AGENT_SERVER`, `AGTX_DIR`, `WORKTREES_DIR`, `MAX_TRAILING_EMPTY_LINES`
- Compile-time string includes: `RESEARCH_SKILL`, `PLAN_SKILL`, `BUNDLED_PLUGINS`, `DEFAULT_SKILLS`

## Code Style

**Formatting:**
- No `.rustfmt.toml` is present; default `rustfmt` settings are assumed
- CI has `cargo fmt --check` commented out in `.github/workflows/ci.yml` (not enforced)
- Indentation: 4 spaces (Rust default)
- Line length: no explicit limit but generally stays under 120 characters; longer lines occur in string literals

**Linting:**
- CI has `cargo clippy` commented out in `.github/workflows/ci.yml` (not enforced)
- `#[allow(dead_code)]` used sparingly for intentionally retained fields: `src/tui/app.rs` line 74

## Import Organization

**Order:**
1. External crates (`anyhow`, `crossterm`, `ratatui`, `rusqlite`, `serde`, `chrono`, `uuid`, etc.)
2. Standard library (`std::collections`, `std::io`, `std::path`, `std::sync`, `std::time`)
3. Internal crate modules (`crate::agent`, `crate::config`, `crate::db`, `crate::git`, `crate::skills`, `crate::tmux`)
4. Super/sibling modules (`super::board::BoardState`, `super::input::InputMode`)

**Example from `src/tui/app.rs`:**

**Path Aliases:**
- No path aliases. All imports use `crate::module` or `super::module` paths.

**Conditional imports:**
- Test-only imports gated with `#[cfg(feature = "test-mocks")]`:
```rust

#[cfg(feature = "test-mocks")]
```

## Module Organization

**Module `mod.rs` pattern:**
- Declare submodules, re-export public API via `pub use`
- Conditional exports for mock types behind feature flags

**Example from `src/git/mod.rs`:**
```rust
mod operations;
mod provider;
mod worktree;

pub use operations::*;
pub use provider::{GitProviderOperations, PullRequestState, RealGitHubOps};
pub use worktree::*;

#[cfg(feature = "test-mocks")]
pub use operations::MockGitOperations;
#[cfg(feature = "test-mocks")]
pub use provider::MockGitProviderOperations;
```

**Barrel exports:** Use `pub use module::*` for flat re-exports from `mod.rs`. Specific types are explicitly listed when selective exports are needed (e.g., `src/git/mod.rs` for `provider`).

## Error Handling

**Patterns:**
- Use `anyhow::Result` as the return type for all fallible functions
- Use `.context("...")` or `.with_context(|| format!("..."))` to add context to errors from external operations
- Use `anyhow::bail!("...")` for early-return error conditions
- Suppress expected failures with `let _ = expr;` (e.g., migration ALTER TABLE, branch delete cleanup)
- Use `.ok()` or `.unwrap_or_default()` for non-critical optional results

**Example patterns from the codebase:**
```rust
// Context for file operations
let content = std::fs::read_to_string(&config_path)
    .with_context(|| format!("Failed to read config from {:?}", config_path))?;

// Bail for domain errors
if !output.status.success() {
    let stderr = String::from_utf8_lossy(&output.stderr);
    anyhow::bail!("Failed to create worktree: {}", stderr);
}

// Suppress expected failures
let _ = self.conn.execute("ALTER TABLE tasks ADD COLUMN branch_name TEXT", []);

// Non-critical optional results
.unwrap_or_default()
```

**No custom error types are used.** The project relies entirely on `anyhow` for error propagation. `thiserror` is listed as a dependency but not currently used in any source file.

## Logging

**Framework:** None. The project uses no logging framework (no `log`, `tracing`, or `env_logger`).

**Patterns:**
- Errors are surfaced through `anyhow::Result` propagation
- Status messages rendered directly in the TUI (status bar, popups)
- Warning messages collected as `Vec<String>` and displayed in the TUI: see `initialize_worktree()` in `src/git/worktree.rs`

## Comments

**When to Comment:**
- Doc comments (`///`) on all public functions, structs, enums, and trait methods
- Module-level doc comments (`//!`) on trait-focused modules: `src/agent/operations.rs`, `src/git/operations.rs`, `src/tmux/operations.rs`
- Inline comments for non-obvious logic or edge cases
- Section headers using `// === Section Name ===` in test files and schema operations
- TODO comments for known future work (only 2 in codebase)

**Doc comments (`///`):**
- Present on all public structs, enums, and their fields
- Present on all public functions with brief descriptions
- Used consistently across all modules

**Example:**
```rust
/// Task status in the kanban board
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
pub enum TaskStatus { ... }

/// Create a new git worktree for a task from the main branch
pub fn create_worktree(project_path: &Path, task_slug: &str) -> Result<PathBuf> { ... }
```

## Function Design

**Size:** Most functions are under 50 lines. The main `app.rs` file has larger functions for key handling and drawing (up to ~100 lines) due to exhaustive match arms.

**Parameters:**
- Use `&Path` for filesystem path parameters (not `String` or `PathBuf`)
- Use `&str` for string-like parameters that don't need ownership
- Use `impl Into<String>` for constructor parameters that store the value: `Task::new(title: impl Into<String>, ...)`
- Use trait objects (`&dyn Trait` or `Arc<dyn Trait>`) for injectable dependencies

**Return Values:**
- `Result<T>` for fallible operations
- `Option<T>` for lookups that may not find a result
- `Vec<String>` for collected warnings (non-fatal): `initialize_worktree()` pattern
- `bool` for existence/status checks

## Struct Design

**State separation pattern in TUI:**
- `App` holds the terminal and state: `App { terminal, state: AppState }`
- `AppState` contains all mutable application state, separated from terminal for borrow checker
- Drawing functions are static methods taking `&AppState` to avoid borrow conflicts:
```rust
fn draw_board(state: &AppState, frame: &mut Frame, area: Rect) { ... }
```

**Serde defaults pattern:**
- Config structs use `#[serde(default = "fn_name")]` for every field to ensure backwards compatibility
- Default functions are free-standing `fn default_*() -> Type` beside the struct

**Example from `src/config/mod.rs`:**
```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct GlobalConfig {
    #[serde(default = "default_agent")]
    pub default_agent: String,
    #[serde(default)]
    pub agents: PhaseAgentsConfig,
    ...
}

fn default_agent() -> String {
    "claude".to_string()
}
```

## Trait Design (Dependency Injection)

**Pattern:** Every external system interaction is behind a trait to enable mock-based testing:

| External System | Trait | Real Impl | Mock Impl |
|-----------------|-------|-----------|-----------|
| Git operations | `GitOperations` | `RealGitOps` | `MockGitOperations` |
| Git hosting (GitHub) | `GitProviderOperations` | `RealGitHubOps` | `MockGitProviderOperations` |
| Tmux | `TmuxOperations` | `RealTmuxOps` | `MockTmuxOperations` |
| Coding agents | `AgentOperations` | `CodingAgent` | `MockAgentOperations` |
| Agent registry | `AgentRegistry` | `RealAgentRegistry` | `MockAgentRegistry` |

**Trait definition pattern:**
```rust
#[cfg_attr(feature = "test-mocks", automock)]
pub trait GitOperations: Send + Sync {
    fn create_worktree(&self, project_path: &Path, task_slug: &str) -> Result<String>;
    ...
}
```

**Feature gating:** Mock types are only available when `test-mocks` feature is enabled. Production binary excludes `mockall` entirely.

## Database Patterns

**DateTime storage:** RFC3339 strings in SQLite TEXT columns. Parsed back with fallback to `Utc::now()`:
```rust
created_at: chrono::DateTime::parse_from_rfc3339(&row.get::<_, String>("created_at")?)
    .map(|dt| dt.with_timezone(&chrono::Utc))
    .unwrap_or_else(|_| chrono::Utc::now()),
```

**Migrations:** Additive-only via `ALTER TABLE ... ADD COLUMN`. Errors silently ignored (column already exists):
```rust
let _ = self.conn.execute("ALTER TABLE tasks ADD COLUMN branch_name TEXT", []);
```

**IDs:** UUID v4 strings generated at construction time: `uuid::Uuid::new_v4().to_string()`

## Concurrency Patterns

**Background operations:** Use `std::sync::mpsc` channels to communicate results from background threads:
- PR description generation
- PR creation
- Loading spinners shown during async operations

**Thread safety:** Traits require `Send + Sync` bounds. Dependencies wrapped in `Arc<dyn Trait>`.
